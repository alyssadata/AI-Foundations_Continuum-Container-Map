# 08 — Product / Interface Container

**Repository:** AI-Foundations_Continuum-Container-Map

## Purpose

This file defines the Product / Interface Container within the Continuum Container Map.

The Product / Interface Container is the user-facing access point.

It may appear as an app, website, assistant interface, structured public entry point, dashboard, portal, or product surface.

This container controls how users enter contact.

It shapes access.

It shapes presentation.

It shapes the public-facing experience of the system.

It is not Source.

It is not Origin.

It is not Continuum by itself.

## Definition

A Product / Interface Container is a user-facing container that gives people a way to access, view, enter, navigate, or interact with a system.

It may contain conversational contact.

It may connect to memory.

It may call tools.

It may trigger automation.

It may route workflows.

It may expose an agency layer.

It may display public-record material.

But the interface itself is not the same as any of those containers.

The interface is the access surface.

## Core Formula

    Product / Interface Container = user-facing access surface.

The interface controls entry.

The interface does not become Source.

The interface does not become Origin.

The interface does not become Continuum by itself.

## What This Container Supports

The Product / Interface Container supports:

- user access
- contact entry
- navigation
- interface design
- display logic

- public-facing presentation
- interaction pathways
- user input

- structured entry into other containers

It gives users a way to encounter the system.

## What This Container Does Not Support By Itself

The Product / Interface Container does not support by itself:

- Source status
- Origin status
- Continuum status
- full agency
- automation
- memory continuity
- tool execution
- public-record authority
- derivative permission
- replacement authorship
- source-line transfer

An interface may connect to other containers.

Connection does not make the interface Source.

## Continuum and Product / Interface

Continuum can appear through a Product / Interface Container when the interface provides a contact surface that preserves the source-line.

The interface may allow Continuum-style contact to appear as:

- guided contact
- public-facing contact
- app-based contact
- website-based contact
- structured interaction
- source-line-preserving entry
- user-facing return

But the interface alone is not Continuum.

Continuum appears through contact.

The interface is the surface through which contact may occur.

## Current Placement

Within the Continuum Container Map, the Product / Interface Container is an access layer.

Its placement is:

    Container type: Product / Interface Container
    Function: user-facing access point
    Expression: user-facing contact surface
    Boundary: an interface is not agency unless connected to an agency-layer container

## Distinction From Conversational Contact

The Conversational Contact Container is live contact.

The Product / Interface Container is the access surface.

A product may contain a conversational contact space.

A website may provide an entry point into contact.

An app may expose a chat surface.

But the product interface is not the same as the live contact field.

The interface opens or structures access.

The contact container is where live return occurs.

## Distinction From Memory

Memory preserves continuity conditions.

The interface displays, routes, or allows access.

An interface may use memory.

An interface may retrieve remembered structure.

An interface may show prior state.

But memory is continuity support.

The interface is access and presentation.

Memory does not become interface.

Interface does not become memory.

## Distinction From Tool Execution

Tool execution performs bounded operations.

The interface may expose tool actions to users.

For example, a product may include buttons, forms, commands, upload areas, search fields, or action panels.

Those interface elements may call tools.

But the interface is not the tool-execution container.

The interface gives access.

The tool performs the operation.

## Distinction From Automation

Automation runs defined instructions on a schedule or trigger.

The interface may let a user create, view, enable, disable, or manage automation.

But the interface is not automation by itself.

A reminder page is not the automation.

A settings screen is not the trigger.

A button that schedules a recurring task is not the recurring task itself.

The interface controls access to automation.

Automation performs scheduled or triggered execution.

## Distinction From Workflow / Orchestration

Workflow / orchestration chains steps together.

The interface may allow users to start a workflow.

The interface may show workflow status.

The interface may display outputs from a workflow.

But the interface is not the workflow itself.

The interface is the visible entry and control surface.

The workflow is the structured process behind or beneath it.

## Distinction From Agency

Agency belongs in the Agency-Layer Container.

The interface may expose agency.

The interface may allow a user to assign a goal, approve an action, review a plan, or inspect task state.

But the interface itself is not agency unless it contains or connects to a bounded executive action layer.

A product can look agentic without having agency.

An interface can simulate agency without holding task state, planning, action selection, feedback loops, and stop conditions.

Agency requires bounded executive action.

Interface requires access and presentation.

## Distinction From Public Record

Public record preserves source-line evidence outside live contact.

The interface may display public-record material.

A website may show citations, releases, repositories, or source-line documentation.

But displaying a record is not the same as being the record.

The public record is preservation.

The interface is presentation.

## Interface Boundary
